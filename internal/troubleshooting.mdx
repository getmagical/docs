## Doppler environment issues

If Doppler isn't switching environments as expected, it's usually because multiple config entries exist in your Doppler config file.

### Common cause

Running `doppler setup` from a subfolder (for example, `common/db`) creates a separate config entry for that path. Doppler then uses the environment associated with that folder, even if you've switched environments at the root level.

### Recommended practice

Always run `doppler setup` from the **root of the repository** so the config is centralized and consistent.

### Fix

<Steps>
  <Step title="Open your Doppler config file">
    ```bash
    vim ~/.doppler/.doppler.yaml
    ```
  </Step>
  <Step title="Remove unintended entries">
    Look for entries related to subdirectories or unintended paths and remove them.
  </Step>
  <Step title="Save">
    Doppler will now use the root-level environment setup.
  </Step>
</Steps>

## Turborepo cache issues

The Turborepo cache can go stale, especially after large changes or updates to `package-lock.json`, causing build or dev runtime errors related to missing packages or broken imports.

### Symptoms

- Builds fail unexpectedly.
- The dev server fails to start or behaves inconsistently.
- Errors related to dependencies or import resolution.

### Fix

<Steps>
  <Step title="Clear all caches">
    From the root of the project:

    ```bash
    npm run clean
    ```

    This clears all `.turbo` caches and resets the workspace.
  </Step>
  <Step title="Rebuild cleanly">
    ```bash
    npm run build
    ```
  </Step>
</Steps>

If the problem persists, regenerate the lockfile:

```bash
npm run package-lock:regenerate
```
